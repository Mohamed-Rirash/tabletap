## Problems solved

Per-feature bug narratives are in the tracker. Durable environment/infra lessons worth keeping here:

- Playwright IS usable in this environment: `npm install playwright` in a scratch dir picks up the cached Chromium at `~/.cache/ms-playwright` (no download). `/dev/mailbox` (Swoosh local adapter) is real and scriptable too.
- `Repo.prepare_query/3` injects `org_id` into **every** query it runs through the tenant-enforced Repo, including preload sub-queries — `preload: :user` off a tenant-scoped query crashes because `users` has no `org_id` column. Fix: two explicit queries, zipped together.
- **`Repo.put_org_id/1` is ambient process state, not tied to whichever `%Scope{}` you're holding.** Any helper doing Ordering-context reads/writes should re-assert `Repo.put_org_id(scope.org.id)` as its own first line.
- Cross-tenant reads/writes belong in whichever context already owns the underlying table — loop `Tenants.list_org_ids/0` + `Repo.put_org_id/1` per org (`Workers.SweepAbandonedCarts`'s shape). Never reach for `skip_org_id: true` outside `Accounts`/`Tenants`/platform-admin.
- A `phx-trigger-action` form submit is a no-op if clicked before the LiveSocket JS attaches — add `page.waitForLoadState("networkidle")` first in Playwright verify scripts.
- Always boot the real dev server after Repo/Oban/supervision changes; `Oban testing: :manual` hides boot loops from the test suite.
- Root layout has no sitewide `<.flash_group>` — a controller `put_flash` + `redirect` is silently invisible.
- Never hard-match `Decimal.parse/1` on raw user-typed input — returns `:error` for anything non-numeric.
- **`mix compile`/`mix test` can silently start compiling a whole Erlang toolchain from source** if `.tool-versions` names a version `mise` doesn't have cached. Check `asdf list erlang` first; prefix commands with `PATH="/home/rirash/.asdf/shims:$PATH"`.
- `<.money>` with no `locale=` falls back to the app default (`:en`) — differs visibly from a venue-scoped call on the same page. Know which one a row uses before asserting exact formatted text.
- **`timestamps(type: :utc_datetime)` (second precision) means two rows in the same wall-clock second are a real tie for `order_by: [desc: :inserted_at]`.** Always add a secondary sort key. When a test also needs to assert exact order, sort its own "expected" list by the *identical* composite key (`{DateTime.to_unix(inserted_at), id}`, not just `id` alone) — Feature 17's own tie-break test had exactly this bug (assumed a tie, broke when timestamps happened to differ), fixed in Feature 18 part 4a.
- Elixir requires every `def handle_event/3` (or same-name/arity function) clause to be contiguous — a `defp` between two clauses is a compile error under `--warnings-as-errors`. Recurred 3+ times this project; always append new private helpers *after* the full clause group.
- A brand-new venue has `charges_enabled: false` and `pay_at_counter_enabled: false` — a verify script must flip `pay_at_counter_enabled` on before a guest can check out.
- Phoenix's `<.button>` component doesn't render `type="submit"` — a raw `<button type="submit">` does. Check which one a template uses before writing a Playwright/test selector.
- **A raw `Date`/`DateTime` struct can't go into a `~p"...?#{[key: value]}"` verified-route query string** — `Phoenix.Param` has no impl for it, raises `ArgumentError` ("structs expect an :id key"). Always `Date.to_string/1` first. Bit every new Analytics LiveView's CSV-export link and range picker this session.
- `Enum.sort_by/3`'s `{:asc/:desc, Module}` sorter works for any module exposing `compare/2` returning `:lt/:eq/:gt` — confirmed both `Money.compare/2` (same-currency only) and `Decimal.compare/2` satisfy this, so `{:desc, Money}`/`{:desc, Decimal}` are safe, idiomatic sorters throughout Feature 18's Analytics code instead of hand-rolled comparators.
- **This session hit a live concurrent-edit collision**: another session was working on Feature 19 (Subscriptions & Platform Admin) in the *same checkout* at the same time. Several of my own uncommitted files got silently reverted to their last-committed state mid-session (lost ~30 min of Analytics/DashboardLive work, had to redo from conversation history). Lesson: if a file you just wrote comes back reading like an older version, don't assume you misremembered — check `git status`/`git diff` immediately, since concurrent writers to the same working tree are a real failure mode here, not a hypothetical. When it happens, stop and ask the user rather than guessing whether to overwrite.
- When two sessions share a working tree and a shared file (router.ex, layouts.ex) ends up with both sessions' uncommitted hunks mixed together, ask the user how to handle committing it (bundle together vs. hold out) rather than guessing — this user's call was "bundle together, don't lose either session's work."
- **Two Ecto schemas can't reference each other at compile time even indirectly** — `Reports.subscribe/3` pattern-matches `%ReportSubscription{}` (a struct literal), and `ReportSubscription`'s own schema tried `Ecto.Enum, values: Reports.report_types()` (a function call into `Reports`), and the two files deadlocked compiling each other. Fix: inline the literal atom list in the schema instead of calling back into the context module — a small, deliberate duplication rather than a real coupling.
- **`Repo.prepare_query/3`'s auto-injected `org_id` filter applies to *every* nested preload query, including ones against schemas that were never meant to be tenant-scoped** (`Org` itself — no `org_id` column, an org *is* the tenant; `User` — no `org_id` column, a user can span orgs via multiple memberships). Any preload or join that touches `Org`/`User` from tenant-scoped code needs its own two-query fetch with explicit `skip_org_id: true` (`Tenants.list_memberships/2` is the canonical example — read its own comment before reinventing this).
- **Swoosh test mailbox is FIFO and un-filtered at receive time** — `assert_email_sent(to: ...)` is `assert_received {:email, email}` (grabs the oldest `{:email, _}` message) *then* checks params against whichever one it got. A test fixture that itself sends email (`AccountsFixtures.user_fixture/1` does, via `deliver_login_instructions`) pollutes the mailbox ahead of the email you actually want to assert on. Drain the mailbox (`receive do {:email, _} -> ... after 0 -> :ok end`, looped) right before triggering the send you're testing.
- **`Swoosh.Attachment.new({:data, binary}, filename: ..., content_type: ...)` attaches in-memory content** (no temp file needed) — confirmed via `deps/swoosh/lib/swoosh/attachment.ex`, used for emailing a CSV built in-process rather than round-tripping through disk or an authenticated download link.
- **This environment's test DB connection pool (`pool_size: schedulers_online() * 2` = 8 on this 4-core box) is smaller than some existing concurrency-hammer tests' own concurrent-task count (e.g. a "10 concurrent checkouts" test in `checkout_test.exs`).** Intermittent `DBConnection.ConnectionError`/"could not checkout the connection" failures in `checkout_test.exs`/`tenants_test.exs` are pre-existing environmental flakiness from this mismatch, not a real logic bug — reproduced across several `mix test` runs, confirmed unrelated to files touched by Feature 20 Commit 4. If this needs fixing, the target is `config/test.exs`'s `pool_size` or those tests' own concurrency count, not anything in Feature 20.
- **`phx-hook` only initializes on elements inside a LiveView's own socket-managed DOM subtree.** `root.html.heex` is the static shell around `{@inner_content}` — a `phx-hook` placed directly in the root layout (outside any `<.live_view>` root) never fires, since LiveView only discovers/binds hooks while diffing a View's own container. Anything living in the root layout itself (offline banner, install-prompt button) needs plain vanilla JS wired at page load in `app.js` instead, operating on the element by id — same shape as the pre-existing theme-toggle `<script>` and topbar wiring already in that file.
- **Manually verifying cross-node `Phoenix.PubSub` locally (two `elixir --sname` BEAM nodes) is genuinely doable and worth doing — but budget a generous receive-timeout (60–90s+) on the "listener" node, not a tight one (~15–30s).** The actual `Node.connect/1` + `:pg` group sync always completed in well under a second every time it was checked directly (`:pg.get_members/2`) — every "no message received" failure was purely because composing/running the second node's script (via several sequential tool calls: write file, sleep, run) ate more wall-clock time than the listener's own short timeout, not a real clustering defect. Confirmed by directly RPC-querying the listener node's own `Node.list()`/`:pg.get_members/2` mid-test — cluster membership was always correct; only the timing of the test harness itself was wrong.
- **A second Ecto repo pointed at the same physical database (`Tabletap.ObanRepo`, kept separate from `Tabletap.Repo` specifically so Oban's own bookkeeping tables stay outside tenant `org_id` scoping) needs its OWN `Ecto.Adapters.SQL.Sandbox.mode(_, :manual)` call in `test_helper.exs` and its own `Sandbox.start_owner!/2` in `DataCase` — the sandbox checkout for one repo module does not cover a second repo module even against the identical database.** Without it, every `Oban.insert/1` in every test ever run commits for real instead of rolling back at test end — silent, invisible until something asserts `refute_enqueued(worker: X)` with no `args:` filter (broad enough to catch stray rows from unrelated tests/runs). Found via 3375 accumulated stray rows in `tabletap_test.oban_jobs`. Any future second-repo addition in this codebase needs the same two-line treatment from day one.
- **Oban's `unique:` option has two easy-to-miss requirements, both found by a test that asserted the actual enqueued job's args (not just that `Oban.insert/1` returned `{:ok, _}`):** (1) `keys:` must be a list of **atoms** even when the worker's own `args` map uses string keys throughout (every worker in this codebase does) — passing string keys doesn't warn, it makes the changeset invalid and `Oban.insert/1` returns `{:error, changeset}`. (2) `fields:` must explicitly include `:worker` if you also list `:args` — omitting it means Oban compares args *across every worker*, so a new job can get silently treated as a duplicate of a same-shaped-args job belonging to a completely different worker (`SendPush` colliding with an unrelated `AssignWaiter` job that happened to share an `order_id` key, both because `Ordering`'s own job-arg convention of `{"order_id" => ..., "org_id" => ...}` is reused loosely across several different workers).
- **A named GenServer's own ETS table (`RateLimiter`, and now `Payments.GatewayHealth`) is process-independent shared state across the whole test run — asserting on its cumulative value from inside an `async: true` test file is racy against every other concurrently-running async test that also mutates it.** ExUnit runs the whole `async: true` pool concurrently and only runs `async: false` modules afterward (never interleaved), so the fix is a small, dedicated `async: false` test file that resets the shared state in its own `setup` and asserts on it directly — never bolted onto an existing `async: true` file. `test/tabletap/payments/gateway_health_test.exs`/`gateway_health_integration_test.exs`/`test/tabletap_web/live/manager/gateway_degraded_banner_test.exs` are the pattern to copy for any future shared-ETS-backed feature. **By contrast, DB-backed state (Ecto Sandbox) genuinely is safe under `async: true`** — each test's own sandboxed transaction is isolated from every other test's, so a shared unique slug/constraint across concurrent tests never actually collides (`test/tabletap/ops_test.exs`, `test/tabletap_web/telemetry_test.exs`'s Oban-queue-depth test). Don't reflexively reach for `async: false` for every "shared resource" concern — the real dividing line is process-independent memory (ETS/GenServer state) vs. sandboxed DB state.
- **`Repo.get_by(Schema, key: val, skip_org_id: true)` silently does NOT do what it looks like — `skip_org_id: true` gets folded into the *clauses* keyword list (the second positional arg), never reaching `get_by/3`'s separate third `opts` argument, so the tenant-scope guard still fires.** Every correct call site in this codebase already wraps clauses in an explicit list: `Repo.get_by(Schema, [key: val], skip_org_id: true)` (see `Accounts.get_user_by_email/1`). Caught building `Tabletap.Ops` (Feature 21) by actually hitting the endpoint live rather than trusting compile success.
- **`Repo.preload(struct, :assoc)` on any association pointing at `Org` or `User` breaks the instant ambient `Repo.put_org_id/1` is set** — `Repo.prepare_query/3` injects `org_id` into preload's own internally-generated query, and neither table has that column (an org *is* the tenant; a user can span orgs). This was already documented in `Tenants.list_memberships/2`'s own comment from an earlier feature, and still bit `Tabletap.Ops.check_order_flow/0` fresh this session — the fix is the same every time: a plain two-query fetch (`Repo.one(from o in Org, where: o.id == ^id, skip_org_id: true)`) and manually attach it (`%{struct | assoc: result}`), never `Repo.preload/2,3` for an `Org`/`User` association from tenant-scoped code.
- **When writing cross-tenant tests that switch `Repo.put_org_id/1` between two orgs in the same test process, always set it back explicitly before creating "my own org"'s fixtures** — it does not reset itself. Cost real debugging time twice in the same commit (Feature 22): once produced a `BadMapError` red herring that looked tenant-related but was actually just destructuring `{order, order_item}` (a test helper's real tuple return, per every *other* caller of it in the same file) as if it were a bare struct; the other produced `Ordering.checkout/2`'s `cart` argument silently being `nil` because `add_to_cart`'s own ambient-scoped re-fetch found nothing under the wrong org. Neither was a real app bug — both were the test's own ambient scope pointed at the wrong org at the moment a fixture was built.
- **A "does org B ever see org A's data" test needs org A and org B's fixtures to be distinguishable in the assertion, not just present** — same-shaped fixtures (identical item name/price across both orgs) make a real leak and correct isolation render identical HTML, so the test can't actually tell them apart. Use a deliberately different, identifying value (a different price, in the POS order-number-collision test) so a real leak would visibly surface in the rendered output instead of passing either way.
