## Known Technical Risks

| Risk | Mitigation |
|---|---|
| Cross-tenant data leak | Three-layer enforcement (Scope, `prepare_query` raise, composite FKs); dedicated audit in feature 22; per-context second-tenant tests |
| Overselling the last portion under concurrent checkouts | Atomic `UPDATE ... WHERE sold_qty < limit_qty` inside the order transaction; concurrency test in CI |
| Webhook delay leaves customer on "confirming…" | Optimistic confirming state + PubSub push the instant the Oban job lands; alert if p95 webhook→placed exceeds 10s |
| Waiter phone dies while assigned | Presence liveness required for assignment; 90s escalation job reroutes; off-shift-with-open-orders forces handoff |
| Stale menu cached by PWA service worker | Network-first strategy for menu/order routes; SW caches only static assets |
| Kitchen tablet browser memory over a 12h shift | Streams (bounded DOM), `TicketTimer` ticks client-side, nightly reload prompt |
| Wallet merchant onboarding (manual, in-person) blocks a venue's launch | Onboarding checklist with the guided "get your merchant account" step; ordering disabled with a clear banner until the verification charge sets `charges_enabled`; we hand-hold the pilot's paperwork |
| WaafiPay callbacks are not retried — a dropped callback could strand a paid order | Reconciliation poller on every pending payment (~30s + pre-expiry sweep) is the guaranteed confirmation path; callback-vs-poller race test in CI |
| WaafiPay unknowns (hostnames, preauth window, payout API, fees) | Flagged UNVERIFIED in research note — confirm directly with WaafiPay before Feature 09; sandbox spike first |
| Ethiopia leg (Chapa/ETB/data-localization Proclamation 1321/2024) | Phase C gated on legal opinion + Chapa test-mode spike + ETB fee-collection decision — Somali cities launch first |
| Unit-conversion mistakes corrupt inventory | Stock stored in base units only; conversions at input boundary with live hint; movements ledger is append-only and re-derivable |
| One venue's traffic spike affecting others | Single BEAM handles it for MVP scale; per-venue PubSub topics; load test at feature 22 defines the real ceiling |
| App down at the moment a customer pays | Stripe retries webhooks up to 72h; idempotent processing; order materializes on recovery — delayed, never lost (see architecture.md Reliability section) |
| Order silently stuck (waiter gone, kitchen missed it) | 90s escalation → claim board; KDS overdue pulse; Oban cron watchdog alerts managers on any non-terminal order past thresholds |
| Whole-node or DB failure | 2-node cluster + LiveView auto-reconnect; managed Postgres failover with PITR; chaos drill in feature 21 proves recovery without order loss |
| ETA wildly wrong early on (no history) | Fall back to static `prep_minutes`; label as "~estimate"; rolling average kicks in after 20 served orders per item |
